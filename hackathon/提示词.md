\# \[02] 前端图形学与 WebGL 架构标准 (Three.js \& GLSL)



\## 2.1 3D 场景与摄影机系统

\- \*\*环境构建\*\*：初始化 `THREE.Scene`。背景设定为极暗的虚空（如 `#020205`）。引入指数级别的雾化效果 `THREE.FogExp2`，让远处的 3D 卡片在视线末端自然消散。

\- \*\*灯光系统\*\*：摒弃全局泛光。采用多盏移动的 `THREE.SpotLight`（聚光灯）模拟演唱会现场的镭射光束。光束需带有微弱的颜色渐变（如从赛博粉过渡到霓虹蓝），并在 3D 空间中按照特定的李萨如曲线（Lissajous curve）轨迹进行持续运动。



\## 2.2 卡片环形阵列与空间几何变换

\- 抓取到的每条演唱会数据，都将映射为一个 3D 空间中的 `THREE.Mesh`。

\- \*\*极坐标分布公式\*\*：卡片需以环形（Carousel）阵列排布。通过基础的空间解析几何进行位置推算：

&#x20; $$x = r \\cos(\\theta)$$

&#x20; $$z = r \\sin(\\theta)$$

&#x20; 其中 $r$ 为圆环半径，$\\theta$ 为每个卡片在极坐标系下的角度偏移量。同时，网格的 `rotation.y` 必须精准指向圆心或朝向摄影机，确保视觉的绝对透视正确。



\## 2.3 自定义着色器与材质 (Custom Shaders)

这是复刻 Active Theory 风格的核心。卡片的材质不可使用普通的 `MeshBasicMaterial`，必须使用 `THREE.ShaderMaterial`，并由自定义的 Vertex Shader（顶点着色器）和 Fragment Shader（片元着色器）接管渲染。

\- \*\*流体扭曲效果 (Liquid Distortion)\*\*：在 Vertex Shader 中，引入基于时间变量 `u\_time` 的 Perlin 噪声函数。当鼠标悬浮或发生转场时，通过计算 UV 坐标的偏移，让海报产生如同水波纹或果冻般的非线性扭曲。涉及到空间扭曲的物理模拟时，应当参考经典的波动方程与二重积分推导，确保动态涟漪的扩散符合流体动力学。

\- \*\*RGB 色差与故障风 (Chromatic Aberration \& Glitch)\*\*：在 Fragment Shader 中分离采样纹理。根据鼠标移动的速度向量 `u\_mouse\_velocity`，对红色通道和蓝色通道的 UV 坐标进行瞬间的撕裂偏移，形成极具冲击力的赛博朋克失真感。

\# \[03] 前端交互、动画与 DOM 层 (GSAP \& Raycaster)



\## 3.1 射线追踪与卡片翻转 (Raycaster \& Flip)

\- \*\*碰撞检测\*\*：在 3D 场景中，监听鼠标的 `mousemove` 和 `click`，通过 `THREE.Raycaster` 从摄影机发射射线，精准捕获当前被选中的演唱会卡片网格。

\- \*\*180° 翻转逻辑\*\*：当卡片被点击时，利用 GSAP (GreenSock) 动画引擎，对选中网格的 `rotation.y` 属性施加 `+= Math.PI` (即 180度) 的缓动动画。卡片的正反面纹理需要在 Shader 中通过判断法线方向（`gl\_FrontFacing`）来分别渲染海报图和背面的票务详细信息。



\## 3.2 滚轮劫持与视差动画

\- 全面禁用 CSS 的 `overflow: scroll`。拦截全局的 `wheel` 事件，提取 `event.deltaY`。

\- 使用 GSAP 的线性插值（Lerp）平滑处理滚轮数据，将其映射为整个 3D 环形场景的旋转角度，或者摄影机在 Z 轴上的推进距离。



\## 3.3 自定义交互光标 (Custom Cursor)

\- 隐藏系统指针 `cursor: none`。利用单独的 Canvas 或固定定位的 DOM 元素绘制一个跟随光标的圆环。

\- 应用 `mix-blend-mode: difference`。当光标滑过高亮的艺人名字时，光标产生磁性吸附（Magnetic Effect），并反转底层像素颜色。

\## 4.1 无头浏览器隐匿与指纹对抗

\- \*\*请求拦截与重放\*\*：通过监听 `page.on('request')` 和 `page.on('response')`，直接在网络层拦截 XHR/Fetch 的返回包，提取纯净的 JSON 演唱会数据，避免高耗时的 DOM 树解析。



\## 4.2 逆向工程与验证码智能突破

\- \*\*参数解密\*\*：对抗前端加密参数（如请求头中的签名字段）时，需利用 AST（抽象语法树）解析思路或直接在浏览器上下文中注入 JS Hook，还原混淆的 JavaScript 执行逻辑。

\- \*\*计算机视觉自动化验证\*\*：若遇到复杂的视觉风控（如滑块缺口识别、空间点选验证码），系统应当预留图像处理与计算机视觉接口。通过加载预训练的轻量级目标检测模型（如 YOLOv8 或 YOLOv11 系列的权重文件），对目标进行端到端的坐标推理与自动化滑动轨迹拟合，实现极高通过率的无人值守闭环。



\## 4.3 数据清洗与标准化

\- 提取字段包括：`artist\_name`（必须去除特殊符号，确保准确）、`tour\_name`、`location`、`show\_time`（统一转换为 ISO 8601 标准时间戳）、`ticket\_status`（映射为标准枚举状态）。

\# \[05] 后端高并发架构与 API 规范 (FastAPI)



\## 5.1 异步非阻塞架构

\- 核心框架采用 \*\*FastAPI\*\*，全面启用 `async/await` 异步处理机制，确保在应对前端高频的轮询或大量渲染请求时，保持极低的延迟（<50ms）。

\- 部署采用 Uvicorn + Gunicorn 工作进程模型，压榨服务器并发性能。



\## 5.2 RESTful API 接口设计

\- `GET /api/v1/concerts`：返回全部演唱会坐标与数据。响应结构必须包含 `data` 数组与 `meta` 元数据。

\- `GET /api/v1/artists/{name}`：针对单体艺人的精准搜索接口。

\- 支持基于 Redis 的接口缓存机制，对于 1 小时内未发生变更的爬虫数据，直接命中缓存，降低数据库 IO 压力。
